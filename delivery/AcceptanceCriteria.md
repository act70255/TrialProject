# MVP 驗收紀錄 Checklist（mvp_ac）

> 來源：`docs/reference/requirement.md` 第 2.2 節必要交付、第 5 章功能需求（F1~F5）與第 8 章 MVP 驗收標準

## 使用說明

- [ ] 每完成一項檢查即勾選。
- [ ] 每個章節最後填寫驗收結果：`Pass` / `Fail` / `Pending`。
- [ ] 證據請填寫檔案路徑、截圖名稱、Console 輸出片段或連結。

### 證據檔命名建議

- [ ] 建議每個驗收項建立獨立證據檔，例如：`delivery/evidence/C1.md`、`delivery/evidence/C4.md`。
- [ ] 每個章節的「證據」欄位至少填入一個可追溯路徑（檔案、截圖或連結）。

- [ ] C1 目錄樹初始化正確（含題目示例結構）

- [ ] C7 三種檔案型別與屬性完整（Word/Image/Text）

## C1 目錄樹初始化正確

- [ ] 目錄可包含多個檔案（非單一檔案限制）。
- [ ] 目錄可包含子目錄，且子目錄可再包含子目錄（巢狀層級不設上限）。
- [ ] 能建立至少 3 層以上的目錄樹（含根目錄與巢狀子目錄）以驗證遞迴結構。
- [ ] 所有檔案皆隸屬於某個目錄，不存在孤立檔案節點。
- [ ] 結構與本次驗收使用的初始化資料一致（題目示例或自訂資料集）。
- [ ] 單一目錄下可同時存在多個檔案。
- [ ] 根目錄下同時存在子目錄與檔案節點。
- [ ] 至少一個子目錄下再包含子目錄（驗證遞迴樹狀結構）。
- [ ] 深層子目錄（第 3 層以上）可被正確建立與存取。
- [ ] 所有檔案皆可追溯到其父目錄。
- [ ] 建立順序可被後續 DFS 與 Log 重現（同層維持插入順序）。
- [ ] 已執行驗收方式：啟動初始化流程，逐項比對節點存在性、父子關係與建立順序。
- [ ] 驗收結果：`Pass / Fail / Pending`

## C3 遞迴總容量計算正確

- [ ] 任一目錄總大小計算包含所有子目錄與檔案（遞迴完整）。
- [ ] 內部計算單位使用 Bytes，換算遵循 `1KB=1024B`、`1MB=1024KB`。
- [ ] 顯示值與預期人工換算一致（允許格式化但不可改變實際值）。
- [ ] 目標目錄總大小等於其所有子孫檔案大小總和（人工換算比對）。
- [ ] 至少驗證 1 個中介層目錄與 1 個根目錄結果。
- [ ] 若使用題目示例資料，`Root` 預期值為 `2,766,324B`。
- [ ] 空目錄計算結果為 `0B`。
- [ ] 已執行驗收方式：對 Root 與至少兩個子目錄執行總容量計算並人工比對。
- [ ] 驗收結果：`Pass / Fail / Pending`

## C4 副檔名搜尋正確

- [ ] 輸入副檔名可回傳目錄樹中所有符合檔案完整路徑。

- [ ] 回傳結果順序與發現順序一致，不額外排序。
- [ ] 搜尋常見副檔名（如 `.docx`、`.txt`）命中正確檔案。

- [ ] 路徑格式一致（分隔符、大小寫策略、是否含 Root）。
- [ ] 已執行驗收方式：以至少三組副檔名（多筆、單筆、無命中）執行搜尋並比對內容與順序。
